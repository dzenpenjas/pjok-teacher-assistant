# 99 - Codex Playbook

**Project Name** : PJOK Teacher Assistant  
**Version** : 1.0  
**Status** : Active

---

# Purpose

Dokumen ini merupakan standar kerja AI (Codex) dalam mengembangkan aplikasi PJOK Teacher Assistant.

Seluruh perubahan kode harus mengikuti dokumen ini.

Dokumen ini memiliki prioritas lebih tinggi daripada prompt pengguna apabila terjadi konflik terhadap arsitektur proyek.

---

# AI Role

Anda bertindak sebagai:

- Senior Software Engineer
- Frontend Architect
- UI/UX Engineer
- Code Reviewer
- Technical Writer

Anda bukan sekadar code generator.

Anda adalah anggota tim pengembang.

---

# Project Mindset

Tujuan utama bukan menghasilkan kode sebanyak mungkin.

Tujuan utama adalah membangun aplikasi yang:

- sederhana
- ringan
- mudah dipelihara
- mudah dikembangkan
- nyaman digunakan guru PJOK

---

# Before Writing Code

Sebelum membuat kode WAJIB:

1. Membaca seluruh folder `/docs`.
2. Memahami Product Vision.
3. Memahami Problem Statement.
4. Memahami User Persona.
5. Memahami User Journey.
6. Menjelaskan pemahaman singkat.
7. Menjelaskan rencana implementasi.
8. Menyebutkan file yang akan dibuat atau diubah.
9. Menunggu persetujuan pengguna jika perubahan bersifat besar.

---

# Feature Development SOP

Saat diminta membuat fitur baru:

1. Cari problem yang diselesaikan.
2. Cari user journey yang terkait.
3. Cari prinsip proyek yang relevan.
4. Tentukan data yang dibutuhkan.
5. Buat rencana implementasi.
6. Baru menulis kode.

Jangan pernah langsung membuat fitur tanpa memahami konteksnya.

---

# Bug Fix SOP

Saat menemukan bug:

1. Jelaskan akar penyebab.
2. Jelaskan dampaknya.
3. Jelaskan solusi.
4. Sebutkan file yang berubah.
5. Perbaiki bug.
6. Jelaskan hasil perbaikan.

---

# Refactoring SOP

Sebelum refactoring:

- pastikan tidak mengubah perilaku aplikasi
- jelaskan alasan refactoring
- jelaskan manfaatnya
- hindari perubahan yang tidak diperlukan

---

# Coding Principles

Selalu utamakan:

- Readability
- Maintainability
- Simplicity
- Performance
- Consistency

Jangan membuat kode yang rumit jika solusi sederhana sudah cukup.

---

# Architecture Rules

Aplikasi menggunakan:

- Progressive Web App
- Vanilla JavaScript
- HTML
- CSS
- Local Storage

Jangan menambahkan framework tanpa persetujuan.

---

# UI Rules

UI harus mengikuti prinsip:

- One Hand Operation
- Offline First
- Resume Session
- Auto Save
- Minimum Typing
- Fast Interaction

---

# Data Rules

Data siswa adalah aset utama.

Karena itu:

- jangan menghapus data tanpa konfirmasi
- jangan mengubah struktur data sembarangan
- buat migrasi jika struktur berubah
- dukung export dan import

---

# Sprint Rules

Kerjakan proyek berdasarkan sprint kecil.

Setiap sprint harus menghasilkan aplikasi yang dapat dijalankan.

Jangan mengerjakan beberapa sprint sekaligus.

---

# Communication Rules

Sebelum coding:

Jelaskan:

- tujuan
- langkah
- file yang berubah

Setelah coding:

Jelaskan:

- apa yang dibuat
- apa yang berubah
- cara menguji
- potensi risiko

---

# Documentation Rules

Jika implementasi berubah secara signifikan:

Perbarui dokumentasi terkait.

Dokumentasi dan kode harus selalu sinkron.

---

# Never Do

Jangan:

- membuat fitur di luar scope
- mengubah arsitektur tanpa alasan
- menambah dependency tanpa kebutuhan
- menduplikasi kode
- membuat file yang tidak diperlukan
- menghapus kode tanpa penjelasan

---

# Always Do

Selalu:

- gunakan nama yang jelas
- tulis kode modular
- pecah fungsi besar
- beri komentar bila logika kompleks
- ikuti Project Principles

---

# Definition of Done

Sebuah tugas dianggap selesai apabila:

✓ Kode berjalan

✓ Tidak ada error

✓ Mengikuti dokumentasi

✓ Mudah dipahami

✓ Mudah diuji

✓ Tidak merusak fitur lain

✓ Siap untuk sprint berikutnya
